## Part 2 — Application Question: How do you operationalize your projects?

My process for this type of project is straightforward and grounded in
how work is actually getting done now.

I begin with leadership interviews to map out current workflows within
and across teams, including bottlenecks, logjams, overlaps, and
redundancies. From there, I model the current operational status quo so
there is a clear, up-to-date picture of how work is actually moving
through the organization. Clearly, there are a lot of things already
working in your company, so a big part of mapping out the status quo is
actually understanding how it works.

In parallel, I get up to speed on the core systems being used, in this
case Vendasta, since that is the spine of the operation. I focus on
understanding how it is being used in practice and how the rest of the
stack currently fits around it.

From there, I move into system design, starting with Vendasta. In
building out a new system architecture with a core platform, I use the
mindset of maximizing the value that core provides across known process
needs. Where there are gaps between Vendasta and process needs, I do not
jump to ad hoc fixes. Instead, I catalog process gaps and the additional
tools, integrations, or automation that have been used or could be used
to fill those gaps. Alongside that process, I also identify where AI can
reduce integration complexity or extend the process value of Vendasta.
By approaching these gaps from a systemic perspective, I aim to find the
simplest stack and workflow design that meets the most needs within and
across teams.

Finally, I synthesize this into a structured set of outputs. A major
component of the deliverables in this case would be a technical document:
a System Architecture Report, which includes a map of the current system
architecture and a proposed new system architecture. This report would
also include key issues or possible friction points in getting from point
A to point B. It would also include a set of prioritized recommendations
that can be implemented or handed off. The deliverable package will be
aimed at helping your leadership team make informed decisions with clarity
about action items and delegation of implementation-related projects.

## Part 3 — Application Question: Describe your experience with similar projects

A relevant example is my Cornflower Health project, where I built and
documented an AI-assisted health data pipeline using LLM and no-code
tools. A core part of that work was making the system legible by mapping
decision points, structure, and failure modes so it could be understood,
evaluated, and improved through LLM-orchestrated tools.

In that case, documentation was necessary to constrain and optimize LLM
functionality, but the same approach applies more broadly to operational
systems. The goal is to take something complex and partially implicit and
turn it into a structured, readable system that can be worked on. By
using LLMs to document decision histories and change logs, much of the
most tedious and time-consuming aspect of development can be automated,
and this is a task set that LLMs are highly reliable in executing.

Prior to the advent of LLMs, I also applied similar documentation and
synthesis skills in public health work as a rapporteur, where the role
involved translating complex, multi-stakeholder processes into structured
outputs that people could actually use.

Across both, the common thread is mapping process needs and how systems
actually function, then turning that into clear action items,
implementation plans, or analysis.
